# KMS на базе Git: тезисы для технической реализации

Краткая спецификация сценария работы с базой знаний и требований к реализации для специалиста по разработке/внедрению.

---

## Назначение документа

Описать модель данных, правила версионирования (Git), формат контента и поведение системы при отдаче статей, поиске и RAG — в объёме, достаточном для проектирования и реализации без дополнительных гипотез.

---

## 1. Роли и права

| Роль | Назначение для реализации |
|------|--------------------------|
| **Reader** | Доступ только к порталу/API; видит только контент со `state: published`, с фильтрацией секций по роли читателя (консультант/эксперт/руководитель). |
| **Author** | Право создавать ветки от `main` и открывать MR; в репо правки только в своих ветках. |
| **Reviewer** | Право одобрять (approve) MR в `main` в рамках своей категории. |
| **Publisher** | Право выполнять merge в `main`. |
| **Admin** | Управление ролями и доступом к репо. |

Роли маппятся на права Git и/или группы AD/ITSM. Реализация должна поддерживать передачу роли пользователя в API отдачи контента и в запрос RAG.

---

## 2. Модель состояния статьи и жизненный цикл

- **Источник истины:** файлы статей в репозитории (Markdown + YAML frontmatter). Состояние статьи задаётся полем **`state`** во frontmatter.
- **Допустимые значения `state`:** `draft` | `review` | `published` | `archived`.
- **Правило публикации:** в ветке `main` считаются опубликованными только документы с `state: published`. Контент из других веток не отдаётся читателям и не индексируется для поиска/RAG.
- **Архив:** при `state: archived` статья исключается из поискового индекса и из индекса RAG; при отдаче по прямому запросу поведение оговаривается отдельно (например, 404 или «снята с публикации»).

Жизненный цикл: draft → review → published; при необходимости переход в archived. Смена состояния — через правку frontmatter в коммите (вручную или скриптом при merge).

---

## 3. Git-workflow и CI

- **Ветка `main`** — единственный источник контента для портала, поиска и RAG. Все изменения в контент попадают в `main` только через merge после ревью (MR/PR).
- **Черновики и ревью:** автор создаёт ветку от `main`, вносит изменения в .md (и при необходимости в .visibility.yaml, ссылки на медиа), пушит и открывает MR в `main`. Reviewer проверяет дифф и апрувит; Publisher выполняет merge.
- **После merge в `main`:** в `main` у статей должен быть `state: published` (выставляется при merge вручную или автоматически по правилу).
- **CI при push в `main`:** обязательно выполнять: (1) сборка артефактов портала/сайта (если есть); (2) обновление поискового индекса по актуальному состоянию репо; (3) переиндексация RAG — парсинг Markdown, разбиение на чанки (например по заголовкам), запись метаданных и эмбеддингов в векторную БД. Индексировать только документы с `state: published`; документы с `state: archived` удалять из индексов.

Реализация должна обеспечивать: разграничение прав на ветки и merge по ролям; запуск пайплайна по событию push в `main`.

---

## 4. Видимость контента по ролям читателя

- **Требование:** в рамках одной статьи часть секций может быть доступна только определённым ролям читателей (например консультант, эксперт, руководитель). Руководитель по соглашению видит все секции.
- **Реализация — один из двух вариантов.**

**Вариант A (разметка в теле статьи):** перед заголовком секции (H2/H3) допускается HTML-комментарий вида `<!-- visibility: role1, role2 -->`. Если комментария нет — секция видна всем. Парсер при сборке/отдаче извлекает границы секций и список ролей из комментария; при отдаче статьи в API передаётся роль пользователя и отфильтрованный список секций.

**Вариант B (отдельный файл видимости):** рядом с `статья.md` допускается файл `статья.visibility.yaml` со структурой: привязка секций по id (slug заголовка) к полю `visible_to` (список ролей). Пример:

```yaml
sections:
  - id: slug-sekcii
    visible_to: [expert, manager]
```

Парсер при сборке/отдаче объединяет контент статьи с данными из .visibility.yaml и фильтрует секции по переданной роли пользователя.

- **Портал/API:** при запросе статьи обязаны принимать параметр роли читателя и отдавать только разрешённые секции.
- **RAG:** при индексации каждый чанк должен быть помечен списком ролей, которым он доступен. При запросе к RAG передаётся роль пользователя; в выборку попадают только чанки, доступные этой роли.

Редактор контента должен позволять задавать видимость по секциям и сохранять её в выбранном формате (комментарий в .md или .visibility.yaml) в рамках того же коммита/MR, что и текст.

## 5. Формат хранения контента: Markdown и HTML

Требуется зафиксировать, в каком формате хранить тело статей в репозитории — Markdown, HTML или комбинация. Учитывается наличие существующего справочника на HTML, используемого для визуализации на сайтах.

### Вариант A: только Markdown

- **Хранение:** тело статьи — Markdown (.md), в начале YAML frontmatter.
- **Плюсы:** удобно для Git (читаемый дифф, мало шума); естественная структура по заголовкам для чанкирования RAG; много редакторов и превью; авторам проще править текст без разметки.
- **Минусы:** сложную вёрстку (таблицы, колонки, виджеты) передавать ограниченно; для визуализации «как на сайте» нужна сборка Markdown → HTML (шаблоны, стили).
- **Справочник:** если справочник сейчас на HTML, он либо остаётся отдельной кодовой базой (HTML-файлы в своём дереве или репо), либо фрагменты справочника рендерятся из Markdown в общий стиль сайта — тогда нужен единый пайплайн «MD → HTML» для и статей, и для переведённого в MD справочника.

### Вариант B: только HTML

- **Хранение:** тело статьи — HTML (.html или фрагмент в .md с HTML-блоками). Метаданные — frontmatter в начале или отдельный YAML.
- **Плюсы:** полный контроль вёрстки и визуализации; единый формат со справочником — один парсер и один способ отображения на сайте; WYSIWYG-редакторы дают вид «как на сайте».
- **Минусы:** дифф в Git менее наглядный; для RAG нужно извлекать текст из HTML (убрать теги, учитывать структуру по заголовкам); выше риск случайной «грязной» разметки и нужна политика санитизации (XSS, допустимые теги).

### Вариант C: Markdown по умолчанию, HTML где нужно

- **Хранение:** основное тело — Markdown; внутри Markdown допускаются сырые HTML-блоки для вставок (таблица, схема, виджет). Стандартные парсеры (например CommonMark) разрешают HTML «как есть» при явном включении.
- **Плюсы:** большинство статей остаётся простым и удобным для диффа и RAG; сложные куски — в HTML без ограничений разметки.
- **Минусы:** правила, где можно HTML (например только определённые теги), и санитизация на стороне сборки/портала обязательны.

### Вариант D: два типа контента в одной системе

- **Статьи KMS** — в Markdown (инструкции, процедуры, текстово-ориентированный контент); парсинг по заголовкам, чанки для RAG, отдача через API с рендером MD → HTML на портале.
- **Справочник** — в HTML (как сейчас), отдельное дерево или префикс в репо; визуализация на сайтах без смены формата; при необходимости поиск и RAG по справочнику — извлечение текста из HTML и индексация с пометкой «справочник».
- Единый репо может содержать оба формата: например `content/articles/` — .md, `content/reference/` — .html; CI и портал обрабатывают оба типа по своим правилам.

### Рекомендация для реализации

- Если **справочник остаётся в HTML** и должен визуализироваться на сайтах без переписывания: целесообразно **вариант D** — статьи в Markdown, справочник в HTML; в спецификации явно описать два типа контента, общие метаданные (frontmatter или аналог для HTML) и раздельные правила парсинга/индексации.
- Если планируется **постепенный переход справочника** в единый формат: выбрать один канонический формат (A или B) и определить миграцию; до её завершения поддерживать оба формата по варианту D.
- В любом случае: зафиксировать в спецификации **допустимый подмножество HTML** (при использовании HTML или вставок HTML в MD), **санитизацию** при отдаче на портал и **правила извлечения текста** для поиска и RAG (по заголовкам, по тегам, игнор скриптов и стилей).

## 6. Медиа (изображения, видео, файлы)

- **Хранение:** медиафайлы хранятся во внешнем хранилище (S3 или аналог, отдача через CDN). В репозитории в статьях хранятся только ссылки или идентификаторы медиа.
- **Формат ссылок в Markdown:** допускается либо полный URL в атрибуте `src`/`href`, либо короткий идентификатор (например `media:путь/к/файлу`), который этап сборки/рендера подменяет на итоговый URL из хранилища. Подпись к изображению (alt) обязательна учитываться для доступности и для индексации в поиск и RAG.
- **Workflow:** загрузка медиа во внешнее хранилище выполняется вне Git (по принятым путям/именам). В статью в том же MR добавляются или изменяются только ссылки в .md. Удаление медиа: удаление ссылки из статьи; удаление файла в хранилище — по процессу (вне рамок данного документа).
- **Поиск и RAG:** индексировать подписи к изображениям (alt), метаданные или текстовое описание видео; бинарные медиа в векторный поиск не включать. Реализация должна позволять боту ссылаться на наличие схемы/видео в статье по этим метаданным.

---

## 7. Структура репозитория и формат статей

- **Размещение:** путь вида `content/категория/локаль/статья-id.md` (или `.html` для контента в формате HTML по п. 5). Рядом с `статья-id.md` допускается `статья-id.visibility.yaml` при использовании варианта B видимости. При варианте D (п. 5) справочник — в отдельном дереве (например `content/reference/`) с расширением .html.
- **Формат статьи:** YAML frontmatter в начале файла (поля как минимум: `state`, при необходимости `id`, `locale`, `title`, `category`, и др. для RAG); далее тело в Markdown или HTML по выбранному формату хранения (п. 5). Ссылки на медиа — по правилам из п. 6.
- **Создание и правка:** единственный способ изменения контента — правка файлов в ветке и merge в `main` через MR. История версий и ревью полностью в Git.

---

## Итог для реализации

Реализовать: (1) хранение контента в Git (Markdown и/или HTML по п. 5, с frontmatter и опциональным .visibility.yaml); (2) CI по push в `main` — сборка, поисковый индекс, RAG-индекс с учётом `state` и видимости по ролям; (3) API/портал отдачи статей с фильтрацией по роли и подстановкой URL медиа; (4) RAG-запросы с фильтром по роли читателя. Медиа — только ссылки в репо, файлы во внешнем хранилище. При двух форматах (статьи + справочник) — раздельные правила парсинга и общие метаданные.
