# Инструкция: репозиторий на GitHub и локальная база знаний

Простое решение по принципам из «Сценарий-работы-и-Git.md»: репозиторий на GitHub, контент в Markdown с frontmatter, просмотр через MkDocs (open source). Редактирование — в любом Markdown-редакторе (Cursor, VS Code, GitHub в браузере).

---

## Часть 1. Создать репозиторий на GitHub

1. **Войти в GitHub** (или зарегистрироваться): https://github.com

2. **Создать новый репозиторий**
   - На главной странице нажать **«New»** (или зелёную кнопку **«New repository»**).
   - **Repository name:** например `kms-base` или `company-knowledge`.
   - **Description:** по желанию, например «База знаний (KMS) на Git».
   - **Public** или **Private** — на ваш выбор.
   - **НЕ** ставьте галочку «Add a README file» (репозиторий создадим из локальной папки).
   - Нажмите **«Create repository»**.

3. **Скопировать URL репозитория**  
   На открывшейся странице будет что-то вроде:
   - HTTPS: `https://github.com/ВАШ_ЛОГИН/kms-base.git`
   - или SSH: `git@github.com:ВАШ_ЛОГИН/kms-base.git`  
   Этот URL понадобится для привязки локальной папки к GitHub.

---

## Часть 2. Локальная папка и первый коммит

Выполнять в терминале (в папке, где лежит ваш проект KMS).

1. **Перейти в папку KMS**
   ```bash
   cd /Users/danilfilippov/Documents/KMS
   ```

2. **Инициализировать Git (если ещё не инициализирован)**
   ```bash
   git init
   ```

3. **Подключить удалённый репозиторий**
   Подставьте свой URL из шага 1.3:
   ```bash
   git remote add origin https://github.com/ВАШ_ЛОГИН/kms-base.git
   ```
   Если репо уже был добавлен и нужно заменить URL:
   ```bash
   git remote set-url origin https://github.com/ВАШ_ЛОГИН/kms-base.git
   ```

4. **Добавить файлы и сделать первый коммит**
   ```bash
   git add .
   git status
   git commit -m "Первоначальная структура KMS: контент, MkDocs, инструкция"
   ```

5. **Отправить в GitHub**
   Ветка по умолчанию может называться `main` или `master`. Если Git создал `master`, переименуйте в `main`:
   ```bash
   git branch -M main
   git push -u origin main
   ```
   При запросе логина/пароля: для HTTPS используйте **Personal Access Token** (GitHub → Settings → Developer settings → Personal access tokens), а не пароль от аккаунта.

После этого репозиторий на GitHub будет содержать все файлы из папки KMS.

## Часть 3. Локальная база знаний и просмотр

В папке KMS уже подготовлены:
- **content/** — статьи в формате Markdown + YAML frontmatter (`state`, `id`, `locale`, `title`, `category` и т.д. по сценарию).
- **mkdocs.yml** — конфиг для MkDocs Material: по этому контенту собирается сайт для просмотра.

### 3.1. Установить MkDocs и тему Material (один раз)

Нужен Python 3. В терминале:

```bash
pip install mkdocs mkdocs-material
```

(или `pip3 install mkdocs mkdocs-material`, если команда `python` не Python 3.)

### 3.2. Запустить просмотр базы знаний локально

В папке KMS выполнить:

```bash
mkdocs serve
```

В консоли появится адрес, обычно: **http://127.0.0.1:8000**. Откройте его в браузере — отобразится список статей и их содержимое.

Остановить сервер: в терминале нажать `Ctrl+C`.

### 3.3. Редактирование статей

- **В редакторе (Cursor / VS Code):** открывайте и правите файлы в `content/.../*.md`. Сохраняйте; при запущенном `mkdocs serve` страница в браузере обновится после перезагрузки (или при включённом авто-обновлении).
- **На GitHub:** в репозитории можно нажать на файл .md → иконка карандаша (Edit) → править в браузере → Commit.

Формат статей: в начале файла — YAML frontmatter (поля `state`, `id`, `locale`, `title`, `category` и др.), далее — текст в Markdown. Добавление новой статьи: новый файл в `content/категория/локаль/статья-id.md` и при необходимости правка `mkdocs.yml` (раздел `nav`), чтобы статья появилась в меню.

### 3.4. Сборка статического сайта (опционально)

Для деплоя куда-либо (хостинг, GitHub Pages):

```bash
mkdocs build
```

Собранный сайт окажется в папке `site/`. Её можно заливать на любой веб-сервер или подключить к GitHub Actions для публикации на GitHub Pages.

## Часть 4. Проверка workflow (ветка → правка → push)

1. Создать ветку для правки:
   ```bash
   git checkout -b edit/refund-flow
   ```

2. Изменить, например, `content/customer-service/ru/refund-flow.md` (сохранить).

3. Закоммитить и отправить ветку:
   ```bash
   git add content/customer-service/ru/refund-flow.md
   git commit -m "Уточнение инструкции по возврату"
   git push -u origin edit/refund-flow
   ```

4. На GitHub открыть репозиторий → появится предложение **«Compare & pull request»** для ветки `edit/refund-flow`. Создать Pull Request (PR) в `main`. После ревью (или без него) выполнить **Merge** — изменения попадут в `main`.

Так проверяется сценарий: контент в Git, правки через ветки и MR/PR, `main` — источник опубликованного контента.

## Кратко

| Шаг | Действие |
|-----|----------|
| 1 | GitHub → New repository → создать репо без README |
| 2 | Локально: `git init`, `git remote add origin URL`, `git add .`, `git commit`, `git push -u origin main` |
| 3 | Установить: `pip install mkdocs mkdocs-material` |
| 4 | Запуск просмотра: `mkdocs serve` → открыть http://127.0.0.1:8000 |
| 5 | Редактировать .md в `content/` в Cursor или на GitHub |

Дальше можно добавить CI (например GitHub Actions) для автоматической сборки и публикации сайта при push в `main`, а также поиск (например плагин для MkDocs) по мере необходимости.
